# LLM Wiki - 个人知识管理系统

## ⚠️ 边界声明（重要）

**本 Skill 处理的是：知识层面的「理解与整理」**
- 把文章内容提炼成知识要点
- 存入 wiki 结构化条目（摘要/概念/实体/对比/概览/综合）
- 分析、消化、归纳文章的核心观点

**走本 Skill 的指令特征：**
- "存到个人维基" / "存入 wiki" / "整理到知识库"
- "提炼文章要点" / "分析这篇文章"
- "帮我消化这篇文章" / "消化一下"
- "做个摘要" / "提取核心观点"
- "存入我的知识库"
- "提炼成笔记"
- "把要点记到 wiki"

**❌ 不要走本 Skill（走 wechat-article-fetcher）：**
- "抓取微信文章" / "下载微信文章"
- "备份公众号文章"
- "保存文章到本地"
- "添加到 CSV 列表"
- "批量下载"

**关键区分：**
| 需求 | Skill |
|------|-------|
| 要文件（下载备份） | wechat-article-fetcher |
| 要知识（提炼存入） | llm-wiki |

## 使用规范

### 文档读取顺序

执行任何 LLM Wiki 操作前，AI **必须**按以下顺序读取规范文档：

1. **首先读取本 SKILL.md**（技能级规范）
2. **然后读取项目 CLAUDE.md**（`~/wiki/CLAUDE.md`，项目级规范，与 wiki/ 同级）
3. **两个文档的规范都要遵守**
   - 如有冲突，以 CLAUDE.md 为准（项目级优先于用户级）
   - 如 CLAUDE.md 未规定的事项，参照 SKILL.md 执行

### 核心理念

这是一个基于 Karpathy LLM Wiki 理念的个人 Obsidian 知识库。LLM Wiki 是一种 AI 增强的知识管理范式：AI 不仅回答问题，更承担"簿记"工作——更新交叉引用、标注矛盾、保持一致性。人类聚焦于素材采集、探索提问和方向把控。

**五大原则：**
1. **原始素材不可变** — 永不修改 raw 文档，只增不减
2. **LLM 拥有 Wiki** — 用户阅读，LLM 负责写入和维护
3. **知识复利** — 每一次摄入和查询都会丰富 Wiki
4. **富维基** — 把有价值的分析沉淀回 Wiki，不要让好答案消失在聊天里
5. **定期除絮** — 保持 Wiki 健康生长的同时维持其健康状态

## 三层架构

### Layer 1: Raw Sources `~/wiki/raw/`
原始文档，不可变。包括：Web clipping、读书笔记、播客摘要、会议纪要、论文。

```markdown
---
id: 2024-04-12-注意力机制概述
source: https://arxiv.org/abs/1706.03762
date: 2024-04-12
tags: [transformer, attention]
---

# Attention Is All You Need - 摘要
（原始内容一字不改）
```

**文件名**：中文，`YYYY-MM-DD-中文标题.md`（**带日期**，记录资料获取时间）

### Layer 2: AI Wiki `~/wiki/wiki/`
AI 综合多个 sources 生成的结构化知识条目。按 Karpathy 类型分类法组织为六种类型。

**类型分类：**

| 类型 | 用途 | 目录 | 标准章节 |
|------|------|------|----------|
| **摘要** | 单篇文章/资料摘要 | `摘要/` | 原文摘要 → 核心要点 → 关键洞察 → 启示 → 相关页面 |
| **实体** | 人、公司、产品 | `实体/` | 基本信息 → 核心特征 → 相关分析 → 相关页面 |
| **概念** | 抽象概念、方法论 | `概念/` | 定义 → 核心要素 → 应用场景 → 相关页面 |
| **对比** | 多方案/对象对比 | `对比/` | 定义 → 评分维度 → 对比案例 → 关键差异 → 结论 → 相关页面 |
| **概览** | 主题/体系全面概览 | `概览/` | 定义 → 核心框架 → 关键概念 → 实践方法 → 经典案例 → 相关页面 |
| **综合** | 跨领域综合、个人思考 | `综合/` | 背景 → 核心设计 → 跨领域关联 → 核心观点 → 相关页面 |

> **跨类型关联说明**：所有类型的最后一节统一为「相关页面」，支持自由关联任何类型（摘要、实体、概念、对比、概览、综合等），打破类型壁垒，形成真正的知识图谱。
>
> **灵活性原则**：标准章节是参考框架而非强制要求。当内容特性与标准结构差异较大时，可灵活调整章节命名和顺序，但应保持逻辑清晰、层次分明。建议优先尝试套用标准结构，仅在确实不适合时再自定义。

> **互联互通规范（重要）**：
> - **跨类型**：从**所有类型**中自由选择关联对象，不限于同类型：
>   - 摘要页可关联：实体、概念、对比、概览、综合、其他摘要
>   - 实体页可关联：概念、对比、摘要、概览、综合、其他实体
>   - 概念页可关联：实体、对比、摘要、概览、综合、其他概念
>   - 对比页可关联：实体、概念、摘要、概览、综合、其他对比
>   - 概览页可关联：所有类型（作为知识枢纽）
>   - 综合页可关联：所有类型（跨领域综合）
> - **质量**：选择真正内容相关的页面，而非随意填充
> - **数量**：不做硬性要求，以内容关联度为准
> - **示例**：投资类摘要应关联相关实体（茅台、腾讯）、核心概念（护城河、商业模式）、概览页（价值投资核心概念）

**Frontmatter 规范：**

```yaml
---
title: 页面标题
type: summary | entity | concept | comparison | overview | synthesis
created: YYYY-MM-DD
updated: YYYY-MM-DD
tags: [tag1, tag2]
confidence: high | medium | low
raw:
  - "[[YYYY-MM-DD-原文标题]]"
related:
  - "[[条目标题]]"
  - "[[另一标题]]"
domains: [投资, 学习]
---
```

**文件名格式：** `类型/标题.md`（**不带日期前缀**），如 `概念/至简三板斧.md`

**概念页示例（互联互通版）：**

> **互联互通要点**：
> - 关联类型：跨类型自由选择（概念↔实体↔摘要↔概览）
> - 关联质量：与内容真正相关
> - 关联数量：不做硬性要求，以内容关联度为准

## 核心操作

### 1. Ingest — 摄入素材

**触发词：** 提供链接/文件/粘贴内容/文件夹

**输入类型及处理方式：**

| 输入类型 | 处理方式 |
|----------|----------|
| URL | WebFetch/smart-web-fetch 抓取 |
| 粘贴文本 | 直接解析 |
| 文件路径 | 读取并解析 |
| 文件夹路径 | 扫描并批量处理 |

**执行步骤：**

**Step 1. 检查是否已存在（必须）**
- 抓取前，先在 `raw/` 目录搜索该 URL（检查 frontmatter 的 `source` 字段）
- **搜索方法**：在 raw 目录中搜索文件内容
  ```powershell
  cd ~/wiki/raw && Select-String -Pattern "source: https://..." -SimpleMatch
  ```
- 如果已存在：
  - 向用户报告：`该文章已保存于 raw/YYYY-MM-DD-标题.md`
  - 询问用户：是跳过 / 重新抓取 / 更新现有文件？
- **如果不存在：继续执行 Step 2**

**Step 2. 识别输入类型**
- URL → WebFetch/smart-web-fetch 抓取
- 文本 → 直接解析
- 文件夹 → 扫描所有 .md/.txt/.pdf 等文件（批量处理时每篇单独检查）

**Step 3. 解析内容**
- 提取标题、主要内容、关键术语
- 记录来源信息（URL、文件名、日期）
- 文件名格式: `YYYY-MM-DD-中文标题.md`
- 文件名中的中文标题必须和提取到的标题一致，不能改动

**Step 4. 存储到 raw/**
- 文件名: `YYYY-MM-DD-中文标题.md`（中文标题来自内容，保留所有标点）
- 添加 YAML frontmatter (id, source, date, tags, accessed)
- **id 必须与文件名完全一致（去掉 .md）**

**Step 5. 创建摘要层（必须）**
- **为每篇 raw 创建一篇摘要页**（`摘要/原文标题.md`）
- 摘要是整篇文章的浓缩，不是简单复制
- 包含核心观点、关键数据、重要结论
- frontmatter 的 `raw:` 指向该 raw 文件
- **关键原则：每篇 raw 必须先有摘要页**

### 用户粘贴内容时的处理方式

| 类型 | 处理方式 |
|------|---------|
| **所有内容** | ✅ 必须先存 raw → 创建摘要 → 提取 |

**原则**：无论外部文章还是用户原创内容，都必须先存 raw，保持规范一致性。

**Step 6. 自动提取知识层（必须）**
- 创建摘要后，**立即全面分析**该摘要内容
- **提取所有符合类型的内容**，不设"重要性"门槛：
  - **概念**：文中定义/提到的所有概念（如"坑口价"、"卖put"）
  - **实体**：文中提到的所有公司、人物、品牌（如"陕西煤业"、"段永平"）
  - **对比**：文中的对比分析（如"买put vs 卖put"）
  - **概览**：文中的体系/方法全面介绍
  - **综合**：跨领域思考或个人总结
- 检查是否已存在：
  - 若已存在 → 更新该页面（添加新 raw 关联）
  - 若不存在 → 创建新页面
- **原则**：单篇摘要即可提取，不等待多篇重复

**Step 7. 提炼知识层（执行提取）**
- **概念页**（`概念/概念名.md`）：提取摘要中所有概念
  - 单篇摘要即可创建概念页
  - 后续其他摘要提到同一概念时，更新该页面
  - frontmatter 的 `raw:` 数组累积所有相关 raw
- **实体页**（`实体/实体名.md`）：提取摘要中所有实体
  - 单篇摘要即可创建实体页
  - 后续更新汇总更多信息
- **对比页**（`对比/对比标题.md`）：提取文中的对比分析
- **概览页**（`概览/概览标题.md`）：提取文中的体系/方法介绍
- **综合页**（`综合/综合标题.md`）：提取跨领域思考

**Step 8. 报告完成**
- 列出本次创建的所有摘要页
- 列出本次创建/更新的所有概念/实体/对比/概览页
- 标注各页面的 `raw:` 关联关系

**Step 9. 更新 index 和 log（必须）**

```
→ 更新 index.md...
✓ index.md 已更新（新增条目索引）
→ 更新 log.md...
✓ log.md 已记录（YYYY-MM-DD Ingest 完成：X个raw、X个摘要、X个概念、X个实体...）
```

**index.md 更新规则：**
- 每创建/更新一个 wiki 条目，同步更新 `index.md` 的相关分类索引
- 按类型分类记录：`摘要/`、`实体/`、`概念/`、`对比/`、`概览/`、`综合/`
- 格式：`[[标题]]`（不带日期）

**log.md 更新规则：**
- 每次操作必须追加一条日志
- 格式：
  ```markdown
  ## YYYY-MM-DD

  ### Ingest
  - 数量：X个raw、X个摘要、X个概念、X个实体...
  - 来源：URL/文件夹路径/粘贴
  - 新增条目：[[标题1�쪽、[[标题2]]...
  - 更新条目：[[标题3]]...
  ```
- 保持按日期倒序排列

**关键检查点**：
- ✅ 每篇 raw 都有对应的摘要页？
- ✅ 是否经过摘要层（而非直接从 raw 拆分）？
- ✅ 概念/实体页：多篇提炼（推荐）或单篇提取（允许）？
- ✅ frontmatter 的 `raw:` 字段完整？

### ❌ 禁止编造规则

**绝对禁止的行为：**
1. **不得虚构 raw 原文中不存在的信息** - 所有内容必须能从 raw 中找到依据
2. **不得基于外部知识填充内容** - 即使你知道相关信息，如果 raw 中没提到，也不能写进 wiki
3. **不得自行补充缺失信息** - 遇到信息缺失时，必须停止并询问用户，而非猜测或编造
4. **不得更改原文含义** - 提取时要忠实于原文，不得曲解或过度解读

**遇到问题时的正确处理：**
```
情况：raw 文件内容异常/缺失/不清晰
❌ 错误：基于已有知识自行填充内容
✅ 正确：向用户汇报问题，询问如何处理

情况：需要提炼概念但信息不足
❌ 错误：从单篇 raw 强行创建概念页
✅ 正确：等待积累更多相关 raw 后再提炼
```

**质量承诺：**
- 所有 wiki 页面内容均可追溯到具体 raw 文件
- 每一条数据、每一个观点都有原文支撑
- 不确定的内容标注 `confidence: low` 并说明原因

```
  → 第二步：从多篇摘要中提炼概念...
  发现重复概念: "安全边际" (出现在A、B、C中)
  ✓ wiki/概念/安全边际.md 已创建 (raw: [文章A, 文章B, 文章C])
  发现重复概念: "商业模式" (出现在D、E中)
  ✓ wiki/概念/商业模式.md 已创建 (raw: [文章D, 文章E])

  发现实体: "腾讯" (出现在A、F中)
  ✓ wiki/实体/腾讯.md 已创建 (raw: [文章A, 文章F])
  完成: 12个 raw → 12个摘要 + 4个概念 + 2个实体
```

### 2. Query — 查询知识

**触发词：** 提问

**⚠️ 关键检查点：** Query 开始前，确认用户是想「查 wiki」还是「让你搜外部资料回答」—— 如果 wiki 没有相关内容才搜外部。

**执行步骤：**

1. **理解问题**
   - 提取关键概念和术语
   - 识别问题类型（事实/解释/分析/建议）

2. **读取索引定位**（关键步骤）
   - 首先读取 `index.md` 索引文件
   - 根据索引文件的分类结构（摘要/实体/概念/对比/概览/综合）定位可能相关的条目
   - 不要直接用 grep 关键词搜索 wiki 文件夹

3. **读取相关条目并遍历内嵌引用**
   - 根据索引定位的结果，读取相关 wiki 条目内容
   - 沿着条目中的 `[[wikilinks]]` 继续深入阅读相关页面，将知识图谱串联起来
   - 综合多个条目及其引用内容回答

4. **补充来源**
   - 当 wiki 条目内容不足以回答时，引用 raw/ 中的原始素材
   - 提供可核查的引用

5. **标注不确定性**
   - confidence 等级 (high/medium/low)
   - 明确知识空白区域

6. **富维基触发**
   - 当答案产生了有价值的产物（比较、分析、联系）时，提示用户：「这似乎值得保留。把它归档为维基页面？」
   - 如果用户确认 → 先在 `raw/` 添加一条 AI 合成笔记（引用原对话） → 再执行完整 Ingest 流程（创建摘要 → 提取知识层 → 更新 index 和 log）

**Query 完成时的 log 更新：**
```
→ 更新 log.md...
✓ log.md 已记录（YYYY-MM-DD Query 完成：查询「XXX」）
```

#### 富维基机制
**触发词：** 「存到维基」「这个值得记」「反馈到 wiki」

**判断标准：** 这个回答是否解决了一个以后还会问的问题？

**执行步骤：**

1. 将本次回答的完整内容写入 `raw/` 目录，创建一个 AI 合成笔记（frontmatter 的 `source` 字段注明对话来源）
2. 执行完整的 Ingest 流程：创建摘要 → 提取知识层（概念/实体/对比/概览/综合）→ 更新 index 和 log

**输出示例：**

```
用户: 老唐的三板斧是什么？
AI: [综合回答，标注 confidence 和来源]

用户: 存到维基
AI: → 将 AI 合成笔记写入 raw/...
✓ raw/2026-04-16-富维基-老唐三板斧分析.md 已创建
→ 执行 Ingest 流程...
✓ wiki/摘要/富维基-老唐三板斧分析.md 已创建
✓ wiki/概念/至简三板斧.md 已更新（新增关联）
→ 更新 index.md...
✓ index.md 已更新
→ 更新 log.md...
✓ log.md 已记录（YYYY-MM-DD 富维基完成：新增[[老唐三板斧分析]]）
```

### 4. Lint — 维基体检

**触发词：** 「维基体检」「lint」「检查维基」

**执行步骤：**

1. **检查断链**
   - 扫描所有 `[[wikilinks]]`
   - 验证目标文件在 wiki/ 目录存在

2. **发现矛盾**
   - 同一概念在不同条目的说法对比
   - 事实性错误检测
   - 标注矛盾程度（严重/轻微）

3. **识别知识空白**
   - 相关主题缺失检测
   - 过时信息标注（超过 30 天未更新）
   - 孤儿页面检测（无引用链的页面）

4. **输出报告**
   - 问题列表（类型、位置、建议修复）
   - 优先级排序

**执行频率：** 每两周一次，或积累 20+ 新素材后集中 lint。

**输出示例：**

```
两周后，用户: 维基体检
AI:
  ✓ 断链: 无
  ⚠ 矛盾: wiki/研究/注意力机制.md 与 wiki/研究/transformer架构.md
          对「multi-head attention 头数」描述不一致 (8 vs 16)
  ⚠ 孤儿: wiki/学习/强化学习.md 无引用
  ⚠ 过时: 语言模型.md 超过 30 天未更新
  建议: 确认头数；删除或保留强化学习？
→ 更新 log.md...
✓ log.md 已记录（YYYY-MM-DD Lint 完成：发现3个问题，断链0个）
```

## 目录结构

```
~/wiki/
├── raw/                              # 原始素材（不可变）
│   └── 2026-04-12-至简三板斧.md
├── wiki/                            # AI 合成 Wiki 条目（按类型分类，**不带日期**）
│   ├── 摘要/                         # 单篇文章摘要
│   │   └── 最棒的教学现场.md
│   ├── 实体/                         # 人、公司、产品
│   │   └── 腾讯.md
│   ├── 概念/                         # 抽象概念、方法论
│   │   ├── 至简三板斧.md
│   │   └── 安全边际.md
│   ├── 对比/                         # 对比分析
│   │   └── 酱香型vs浓香型.md
│   ├── 概览/                         # 主题概览
│   │   └── 老唐投资体系.md
│   └── 综合/                         # 跨领域综合
│       └── 价值投资写作设计.md
├── CLAUDE.md                         # Schema 规范（与 wiki/ 同级）
├── index.md                          # 内容索引
├── log.md                            # 操作日志
└── .obsidian/                        # Obsidian 配置（可选）
```

**Wiki 条目命名：**
- **文件名：`类型/标题.md`**（如：`概念/至简三板斧.md`）—— **不带日期前缀**（知识是永恒的）
- **条目间通过 `[[标题]]` 相互链接**（不带日期）
- 类型由 AI 在 Ingest 时根据内容性质自动确定（摘要/实体/概念/对比/概览/综合）
- 创建时间通过 frontmatter 的 `created` 字段记录
