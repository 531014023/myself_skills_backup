# CLAUDE.md - LLM Wiki Schema（项目初始化模板）

> 本文件为 LLM Wiki 项目的 Schema 初始化模板。当用户使用 llm-wiki skill 初始化新项目时，将本文件内容复制为目标项目的 `CLAUDE.md`。
>
> 本模板基于当前最新实践更新，包含了完整的日常运行规范（Ingest/Query/Lint 三大核心操作、格式规范、禁止规则）。

---

## 核心理念

这是一个基于 Karpathy LLM Wiki 理念的个人 Obsidian 知识库。LLM Wiki 是一种 AI 增强的知识管理范式：AI 不仅回答问题，更承担"簿记"工作——更新交叉引用、标注矛盾、保持一致性。人类聚焦于素材采集、探索提问和方向把控。

**五大原则：**
1. **原始素材不可变** — 永不修改 raw 文档，只增不减
2. **LLM 拥有 Wiki** — 用户阅读，LLM 负责写入和维护
3. **知识复利** — 每一次摄入和查询都会丰富 Wiki
4. **富维基** — 把有价值的分析沉淀回 Wiki，不要让好答案消失在聊天里
5. **定期除絮** — 保持 Wiki 健康生长的同时维持其健康状态

## 禁止编造规则

**绝对禁止的行为：**
1. **不得虚构 raw 原文中不存在的信息** - 所有内容必须能从 raw 中找到依据
2. **不得基于外部知识填充内容** - 即使你知道相关信息，如果 raw 中没提到，也不能写进 wiki
3. **不得自行补充缺失信息** - 遇到信息缺失时，必须停止并询问用户，而非猜测或编造
4. **不得更改原文含义** - 提取时要忠实于原文，不得曲解或过度解读

**遇到问题时的正确处理：**
```
情况：raw 文件内容异常/缺失/不清晰
❌ 错误：基于已有知识自行填充内容
✅ 正确：向用户汇报问题，询问如何处理

情况：需要提炼概念但信息不足
❌ 错误：从单篇 raw 强行创建概念页
✅ 正确：等待积累更多相关 raw 后再提炼
```

**质量承诺：**
- 所有 wiki 页面内容均可追溯到具体 raw 文件
- 每一条数据、每一个观点都有原文支撑
- 不确定的内容标注 `confidence: low` 并说明原因

## Confidence 评级

| 级别 | 含义 | 使用场景 |
|------|------|----------|
| **high** | 高置信度 | 多个 sources 交叉验证，或作者本人确认 |
| **medium** | 中等置信度 | 单一来源但可信度较高，或主流观点 |
| **low** | 低置信度 | 单一来源、推断、待验证的观点 |

## 类型分类（Karpathy 方法）

### 六类知识条目

| 分类 | 定义 | 内容示例 |
|------|------|----------|
| **摘要** | 单篇文章/资料的精华摘要 | 教学案例、读书笔记，会议纪要 |
| **实体** | 具体的人、公司、产品，品牌 | 腾讯、茅台、老唐、巴菲特 |
| **概念** | 抽象概念、方法论、原则 | 安全边际、商业模式、三板斧 |
| **对比** | 多方案/多对象对比分析 | 酱香vs浓香、茅台vs洋河 |
| **概览** | 某主题/体系的全面概览 | 老唐投资体系、AI视频工作流 |
| **综合** | 跨领域综合、个人思考 | 投资+写作、估值思考 |

### 分类原则

1. **按知识类型分类**（非主题分类）：便于跨主题引用和关联
2. **domains 字段标记主题**：在 frontmatter 中用 domains 标记所属领域
3. **实体/概念页可复用**：如 [[商业模式]] 可被投资、商业分析，品牌研究引用
4. **定期重组**：随内容增长，由 AI 建议调整分类

## 文件格式规范

### Raw 层格式

```yaml
---
id: YYYY-MM-DD-原文标题（与文件名完全一致）
title: 原文完整标题
source: URL 或文件路径
author: 作者名
date: YYYY-MM-DD
type: article | book | paper | video | podcast | note
tags: [tag1, tag2, tag3]
accessed: YYYY-MM-DD（抓取日期）
---

原文内容...
```

**文件名**: `YYYY-MM-DD-原文标题.md`（保留所有标点符号）

### Wiki 层格式

```yaml
---
title: 页面标题
type: summary | entity | concept | comparison | overview | synthesis
created: YYYY-MM-DD
updated: YYYY-MM-DD
tags: [tag1, tag2]
confidence: high | medium | low
raw:
  - "[[YYYY-MM-DD-原文标题]]"
related:
  - "[[条目标题]]"          # 跨类型关联：可关联实体、概念、摘要、概览等
  - "[[另一标题]]"          # 自由选择，以内容关联度为准
  - "[[实体页示例]]"        # 示例：摘要页关联实体
  - "[[概念页示例]]"        # 示例：摘要页关联概念
  - "[[概览页示例]]"        # 示例：摘要页关联概览
domains: [领域1, 领域2]
---

# 页面标题

内容按类型遵循标准章节结构（详见「类型分类」部分）
```

**文件名**: `类型/标题.md`（**不带日期前缀**，知识是永恒的）

**标准章节结构**（按类型）：

| 类型 | 目录 | 标准章节 |
|------|------|----------|
| **摘要** | `摘要/` | 原文摘要 → 核心要点 → 关键洞察 → 启示 → 相关页面 |
| **实体** | `实体/` | 基本信息 → 核心特征 → 相关分析 → 相关页面 |
| **概念** | `概念/` | 定义 → 核心要素 → 应用场景 → 相关页面 |
| **对比** | `对比/` | 定义 → 评分维度 → 对比案例 → 关键差异 → 结论 → 相关页面 |
| **概览** | `概览/` | 定义 → 核心框架 → 关键概念 → 实践方法 → 经典案例 → 相关页面 |
| **综合** | `综合/` | 背景 → 核心设计 → 跨领域关联 → 核心观点 → 相关页面 |

> **说明**：所有类型的最后一节统一为「相关页面」，支持跨类型关联（摘要、实体、概念、对比、概览、综合等均可自由链接）。
>
> **灵活性原则**：标准章节是参考框架而非强制要求。当内容特性与标准结构差异较大时，可灵活调整章节命名和顺序，但应保持逻辑清晰、层次分明。建议优先尝试套用标准结构，仅在确实不适合时再自定义。

### frontmatter 字段说明

| 字段 | 必填 | 说明 |
|------|------|------|
| `title` | ✅ | 页面标题 |
| `type` | ✅ | 六种类型之一 |
| `created` | ✅ | 创建日期 |
| `updated` | ✅ | 最后更新日期 |
| `tags` | ✅ | 标签列表 |
| `confidence` | ✅ | 高/中/低 |
| `raw` | ✅ | 关联的原始资料（带日期）|
| `related` | ✅ | **跨类型关联**（自由选择，见「互联互通规范」）|
| `domains` | 可选 | 所属领域 |

## 互联互通规范

**核心原则**：打破类型壁垒，形成真正的知识图谱。

| 规范项 | 要求 | 说明 |
|--------|------|------|
| **跨类型** | 全类型 | 从摘要、实体、概念、对比、概览、综合**所有类型**中自由选择关联 |
| **质量** | 内容相关 | 选择与当前页面**内容真正相关**的页面，而非随意填充 |
| **数量** | 自由 | 不做硬性要求，以内容关联度为准 |

**跨类型关联示例**：


## 三种核心操作

### 1. Ingest（录入）

**触发条件**: 用户要求保存文章/资料到 Wiki，或主动说"ingest"，或提供 URL/文本/文件夹

**输入类型及处理方式**:

| 输入类型 | 处理方式 |
|----------|----------|
| URL | WebFetch/smart-web-fetch 抓取 |
| 粘贴文本 | 直接解析 |
| 文件路径 | 读取并解析 |
| 文件夹路径 | 扫描并批量处理 |

**完整执行流程**:

#### Step 0. 检查是否已存在（必须）

- 抓取前，先在 `raw/` 目录搜索该 URL（检查 frontmatter 的 `source` 字段）
- **搜索方法**（Windows PowerShell）：
  ```powershell
  cd ./raw && Select-String -Pattern "source: https://..." -SimpleMatch
  ```
- 如果已存在：
  - 向用户报告：`该文章已保存于 raw/YYYY-MM-DD-标题.md`
  - 询问用户：是跳过 / 重新抓取 / 更新现有文件？
- **如果不存在：继续执行 Step 1**

#### Step 1. 识别输入类型

- URL → WebFetch/smart-web-fetch 抓取
- 文本 → 直接解析
- 文件夹 → 扫描所有 .md/.txt/.pdf 等文件（批量处理时每篇单独检查）

#### Step 2. 解析内容

- 提取标题、主要内容、关键术语
- 记录来源信息（URL、文件名、日期）
- 文件名格式: `YYYY-MM-DD-中文标题.md`
- 文件名中的中文标题必须和提取到的标题一致，不能改动

#### Step 3. 存储到 raw/

- 文件名: `YYYY-MM-DD-中文标题.md`（中文标题来自内容，保留所有标点）
- 添加 YAML frontmatter (id, source, date, tags, accessed)
- **id 必须与文件名完全一致（去掉 .md）**

#### Step 4. 创建摘要层（必须）

- **为每篇 raw 创建一篇摘要页**（`摘要/原文标题.md`）
- 摘要是整篇文章的浓缩，不是简单复制
- 包含核心观点、关键数据、重要结论
- frontmatter 的 `raw:` 指向该 raw 文件
- **关键原则：每篇 raw 必须先有摘要页**

#### Step 5. 自动提取知识层（必须）

- 创建摘要后，**立即全面分析**该摘要内容
- **提取所有符合类型的内容**，不设"重要性"门槛：
  - **概念**：文中定义/提到的所有概念（如"坑口价"、"卖put"）
  - **实体**：文中提到的所有公司、人物，品牌（如"陕西煤业"、"段永平"）
  - **对比**：文中的对比分析（如"买put vs 卖put"）
  - **概览**：文中的体系/方法全面介绍
  - **综合**：跨领域思考或个人总结
- 检查是否已存在：
  - 若已存在 → 更新该页面（添加新 raw 关联）
  - 若不存在 → 创建新页面
- **原则**：单篇摘要即可提取，不等待多篇重复

#### Step 6. 报告完成

- 列出本次创建的所有摘要页
- 列出本次创建/更新的所有概念/实体/对比/概览页
- 标注各页面的 `raw:` 关联关系

#### Step 7. 更新 index 和 log（必须）

```
→ 更新 index.md...
✓ index.md 已更新（新增条目索引）
→ 更新 log.md...
✓ log.md 已记录（YYYY-MM-DD Ingest 完成：X个raw、X个摘要、X个概念、X个实体...）
```

**index.md 更新规则：**
- 每创建/更新一个 wiki 条目，同步更新 `index.md` 的相关分类索引
- 按类型分类记录：`摘要/`、`实体/`、`概念/`、`对比/`、`概览/`、`综合/`
- 格式：`[[标题]]`（不带日期）

**log.md 更新规则：**
- 每次操作必须追加一条日志
- 格式：
  ```markdown
  ## YYYY-MM-DD

  ### Ingest
  - 数量：X个raw、X个摘要、X个概念、X个实体...
  - 来源：URL/文件夹路径/粘贴
  - 新增条目：[[标题1]]...
  - 更新条目：[[标题2]]...
  ```
- 保持按日期倒序排列

### 2. Query（查询）

**触发条件**: 用户向 Wiki 提问，要求基于已有知识回答

**⚠️ 关键检查点：** Query 开始前，确认用户是想「查 wiki」还是「让你搜外部资料回答」—— 如果 wiki 没有相关内容才搜外部。

**执行流程**:
1. **理解问题** - 提取关键概念和术语，识别问题类型（事实/解释/分析/建议）
2. **读取索引** - 先读取 `index.md` 索引文件，根据索引分类（摘要/实体/概念/对比/概览/综合）定位可能相关的条目
3. **读取相关条目并遍历内嵌引用** - 根据索引定位，读取相关 wiki 条目内容；同时沿着条目中的 `[[wikilinks]]` 继续深入阅读相关页面，将知识图谱串联起来，综合多个条目及其引用回答
4. **补充来源** - 当 wiki 条目内容不足以回答时，引用 raw/ 中的原始素材
5. **标注不确定性** - confidence 等级 (high/medium/low)，明确知识空白区域
6. **富维基触发** - 当答案产生了有价值的产物（比较、分析、联系）时，提示用户：「这似乎值得保留。把它归档为维基页面？」
   - 如果用户确认 → 先在 `raw/` 添加一条 AI 合成笔记（引用原对话） → 再执行完整 Ingest 流程（创建摘要 → 提取知识层 → 更新 index 和 log）

**输出形式**:
- Markdown 回答
- 对比表格
- 可视化图表
- 可直接写回 Wiki 沉淀

### 3. Lint（检查）

**触发条件**: 用户要求"检查 Wiki"、"体检"，或定期自动执行

**执行步骤**:

1. **检查断链**
   - 扫描所有 `[[wikilinks]]`
   - 验证目标文件在 wiki/ 目录存在

2. **发现矛盾**
   - 同一概念在不同条目的说法对比
   - 事实性错误检测
   - 标注矛盾程度（严重/轻微）

3. **识别知识空白**
   - 相关主题缺失检测
   - 过时信息标注（超过 30 天未更新）
   - 孤儿页面检测（无引用链的页面）

4. **输出报告** - `lint-report-YYYY-MM-DD.md`
   - 问题列表（类型、位置、建议修复）
   - 优先级排序

**执行频率：** 每两周一次，或积累 20+ 新素材后集中 lint。

**检查项**：
- [ ] 断链检查（内部链接是否有效）
- [ ] 孤立页面（无入链的页面）
- [ ] 内容矛盾（不同页面间的冲突）
- [ ] 过时内容（被新资料推翻的说法）
- [ ] 缺失的重要概念页面
- [ ] 格式合规性（frontmatter 完整性）

**Lint 输出示例：**

```
两周后，用户: 维基体检
AI:
  ✓ 断链: 无
  ⚠ 矛盾: wiki/概念/至简三板斧.md 与 wiki/概念/安全边际.md
          对「估值标准」描述需统一
  ⚠ 孤儿: wiki/实体/古井贡.md 无引用
  ⚠ 过时: 商业模式.md 超过 30 天未更新
  建议: 确认估值标准；补充古井贡引用或合并
→ 更新 log.md...
✓ log.md 已记录（YYYY-MM-DD Lint 完成：发现3个问题，断链0个）
```

## 富维基机制

**触发词：** 「存到维基」「这个值得记」「反馈到 wiki」

判断标准：这个回答是否解决了一个以后还会问的问题？

**执行步骤**：
1. 将本次回答的完整内容写入 `raw/` 目录，创建一个 AI 合成笔记（frontmatter 的 `source` 字段注明对话来源）
2. 执行完整的 Ingest 流程：创建摘要 → 提取知识层（概念/实体/对比/概览/综合）→ 更新 index 和 log

**富维基完成时的 log 更新：**
```
→ 将 AI 合成笔记写入 raw/...

→ 执行 Ingest 流程...
✓ wiki/摘要/富维基-标题.md 已创建
✓ wiki/概念/...（或其他类型）已创建
→ 更新 index.md...
✓ index.md 已更新
→ 更新 log.md...
✓ log.md 已记录（YYYY-MM-DD 富维基完成：新增[[标题]]）
```

## 命名规范

| 位置 | 格式 | 示例 | 说明 |
|------|------|------|------|
| **raw** | `YYYY-MM-DD-原文标题.md` | `2026-04-12-至简三板斧.md` | 带日期，记录资料获取时间 |
| **wiki** | `类型/标题.md` | `概念/至简三板斧.md` | **不带日期**，知识是永恒的 |
| **id** | 与 raw 文件名一致 | `2026-04-12-至简三板斧` | 用于关联引用 |
| **链接** | `[[标题]]` | `[[至简三板斧]]` | 内部链接不带日期 |
